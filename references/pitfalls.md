## 1. Formatting & Layout

### P1: Never set `updateFields`

Setting `updateFields` to `true` causes an annoying "Update fields?" dialog popup every
time the document is opened. The TOC already has cached page numbers via PAGEREF fields;
users can manually right-click and update if needed.

```python
# WRONG — causes popup on every open
settings_elem.append(parse_xml(f'<w:updateFields {nsdecls("w")} w:val="true"/>'))

# CORRECT — omit this entirely
```

### P2: Never insert empty paragraphs for spacing

Empty `doc.add_paragraph()` calls create visible blank lines in Word, especially around
headings. Use `space_before`/`space_after` on paragraph_format instead.

```python
# WRONG
doc.add_paragraph()
heading = doc.add_heading("Section Title", level=1)
doc.add_paragraph()

# CORRECT
heading = doc.add_heading("Section Title", level=1)
heading.paragraph_format.space_before = Pt(12)
heading.paragraph_format.space_after = Pt(6)
```

### P3: TOC field end must be on the last TOC entry

Creating a separate paragraph for the TOC field end, combined with `page_break_before`
on the next heading, produces a blank page between TOC and content.

```python
# WRONG — blank page before next chapter
p_end = doc.add_paragraph()
r_end = p_end.add_run()
r_end._element.append(parse_xml(f'<w:fldChar ... w:fldCharType="end"/>'))

# CORRECT — append to last TOC entry paragraph
r_end_elem = parse_xml(f'<w:r {nsmap_w}><w:fldChar w:fldCharType="end"/></w:r>')
last_toc_p._element.append(r_end_elem)
```

### P6: TOC field begin paragraph must be invisible

The paragraph holding the TOC field begin/instrText/separate codes contains no visible
text, but Word renders it as a blank line with Normal style spacing. Fix by setting line
spacing to exact 1pt (20 twips) and font size to 1pt.

```python
# WRONG — visible blank line below TOC title
p_begin = doc.add_paragraph()
r1 = p_begin.add_run()
r1._element.append(parse_xml(f'<w:fldChar ... w:fldCharType="begin"/>'))

# CORRECT — invisible field begin paragraph
p_begin = doc.add_paragraph()
pPr = p_begin._element.find(qn('w:pPr'))
if pPr is None:
    pPr = parse_xml(f'<w:pPr {nsdecls("w")}></w:pPr>')
    p_begin._element.insert(0, pPr)
pPr.append(parse_xml(
    f'<w:spacing {nsdecls("w")} w:line="20" w:lineRule="exact" w:before="0" w:after="0"/>'
))
r1 = p_begin.add_run()
r1.font.size = Pt(1)
r1._element.append(parse_xml(f'<w:fldChar ... w:fldCharType="begin"/>'))
```

## 3. Figures

### P5: Literature Review figures must come from cited sources

All figures in the literature review must be sourced from published references cited
in the document — never self-generated with matplotlib or similar tools. The caption
must include the citation source.

**How to get figures**: Use figures from verified open-access publisher pages,
institutional repositories, official reports, public datasets, or supplementary files.
Confirm the figure belongs to the cited source before inserting it.

```python
# WRONG
from matplotlib import pyplot as plt
plt.figure(); plt.plot(...); plt.savefig("fig_review_1.jpg")

# CORRECT — source from a verified cited reference, cite in caption
FIGURE_CAPTIONS = {
    "fig_review_1.jpg": "Fig 1 xxx (from verified source title, year[32])",
}
```

### P9: Preserve sub-figure panel references in Translation

The original paper references specific panels (Fig. 1A, Fig. 1B-C), and the Chinese
translation must preserve these. Writing just "Figure 1" without panel labels loses
critical information.

```python
# WRONG
"The workflow and evaluation results are shown in Figure 2."

# CORRECT
"The workflow is shown in Figure 2A, and the evaluation results are shown in Figure 2B-C."
```

**How to determine correct panels**: Read the figure caption — it lists panels (A),
(B-C), (D-E). Match each text claim to the corresponding panel.

### P14: Never use PDF page screenshots as figure images

PDF pages contain article text, headers, page numbers, and captions mixed with the
actual figure. Always use clean figure images from the publisher page, repository asset,
supplementary file, or other verified source.

**Acceptable source patterns**:
- Publisher article page with a dedicated figure download link
- Official report or dataset page with separate image assets
- Supplementary files that clearly identify figure numbers and captions
- Author-provided public repository linked by the cited work

### P17: Literature Review must include 3-5 figures

The document must contain 3-5 figures distributed across different chapters. Each figure
is sourced from cited references with proper attribution in captions.

### P18: Verify figure-content match before insertion

Before inserting any figure:

1. Use the available image viewing tool to inspect the actual image content
2. Read the section text where you plan to insert it
3. Confirm both discuss the same topic
4. Only insert after verification passes

```python
# WRONG — assumed content from paper title, inserted without viewing
if "2.4" in section_title:  # "Method evaluation"
    add_figure(doc, "fig_example.jpg", ...)  # Actually shows a data collection flowchart

# CORRECT — Read image first, then match to correct section
# Read fig_example.jpg -> shows the data collection workflow
# -> This belongs in the data source or methods section
if "2.1" in section_title:
    add_figure(doc, "fig_example.jpg", ...)  # workflow -> data source/methods section
```

## 4. Content & Structure

### P8: Citation placement in Literature Review

Citations `[N]` must be placed at the END of a sentence or clause, not directly after
author names. This is standard Chinese academic writing convention.

```python
# WRONG — citation after author name
"Wang et al.[13] demonstrated..."
"Smith and Lee[1] systematically described..."

# CORRECT — citation at end of sentence/clause
"Quantitative analysis identified a measurable difference between the two groups[13]."
"Scholars systematically described this method as early as the 1970s[1]."

# CORRECT — multiple findings with citations at clause boundaries
"The first study reported changes in the core indicator[18], while the second study
validated the same trend under a different data source[19]."
```

**Key principles**:
1. Citations at sentence/clause end, before punctuation
2. Author names can appear in text but without `[N]` immediately following
3. Multiple sources for one concept: `...damage mechanisms[16,17].`

### P11: Do not force uniform section counts

Section count per chapter should reflect the topic's natural structure, not be
artificially standardized.

```python
# WRONG — forcing every chapter to have exactly 4 sections
"3.3 Data sources and result interpretation"  # unrelated topics forced together
"3.4 Method limitations and future applications"  # unrelated topics forced together

# CORRECT — sections reflect natural boundaries
"3.3 Data sources"
"3.4 Data preprocessing"
"3.5 Main results"
"3.6 Result interpretation"
```

### P12: Term introduction — one document, one scope

Each specialized term or abbreviation should be introduced once per document.

**Format**: `Chinese name (English Full Name, ABBR)`

**Key rules**:
- Abstract + body = same scope. If introduced in abstract, body uses abbreviation only
- Each document type (Translation, Review, Thesis) is an independent scope
- Common abbreviations (DNA, RNA) and chemical formulas need no introduction
- In Translation documents, this formatting rule overrides the original paper's style

### P13: Literature Translation must follow original paper structure

**Rules**:
1. Use the original paper's section order (for example: Abstract, Introduction, Results,
   Discussion, Methods, or whatever structure the source actually uses)
2. Translate headings faithfully — no shortening or paraphrasing
3. Do not add absent sections (Key Points, Conclusion, etc.)
4. Do not remove or merge existing sections
5. Translate every paragraph — no summarizing
6. P13 governs STRUCTURE only. Formatting conventions (P12 term introduction, P8
   citation placement) always override the original paper's style.

## 5. Workflow

### P7: Download original paper PDF for Translation

After generating a translation document, download the source paper PDF to the same
output directory with a descriptive filename (e.g., `Author_et_al_Year_Journal.pdf`).

**Free PDF sources** (priority order):
1. Publisher page or DOI landing page with an open PDF link
2. Institutional repository or official project page linked by the source
3. Author-maintained public page with a direct PDF link
4. If automated download fails, inform the user with the direct URL for manual download

### P15: Large documents must use the split pattern

Do not generate long documents as one single-file script. For thesis body, literature
review, and literature translation, split prose and data into `content_part*.py` files
and keep formatting in the assembly script:

1. `content_part*.py` - prose, references, figure maps, captions, and section constants
2. `content_data.py` - aggregation imports
3. `create_*_docx.py` - formatting helpers and document assembly only

Use parallel workers or subagents when the current runtime supports them. If it does
not, write the content parts sequentially; the split pattern is still mandatory.

Task book and midterm-check forms are shorter table/form documents. They may use a focused
single script unless the user provides unusually long content.

### P16: Mandatory bundled humanizer pass for thesis and review

After drafting thesis body or literature review content, run the bundled humanizer before
final DOCX delivery. Read `skills/humanizer/SKILL.md`, use Academic mode, and preserve
evidence, citations, numerical values, method names, figure references, and claim
strength.

The pass should fix:

1. English syntax carried into Chinese (long nested attributive clauses)
2. Excessive passive voice
3. "De" chain overload (more than 3 consecutive "de" in a phrase)
4. Mechanical connectives (starting every sentence with "However", "Furthermore", "Thus")
5. Translation-ese vocabulary

Use parallel workers or subagents to humanize different sections when available. If the
runtime has no worker mechanism, revise sections sequentially with the same bundled
humanizer instructions, then regenerate the document.

### P17: First successful DOCX generation is never final

The first generated DOCX is only a draft artifact. For thesis body and literature review,
repeat this cycle until clean:

1. Fix prose/content/reference/figure issues in `content_part*.py` or `content_data.py`
2. Run the bundled humanizer if prose changed
3. Run `python3 -m py_compile ...`
4. Regenerate the DOCX
5. Re-check figures, references, humanizer quality, and formatting

Do not deliver merely because `create_*_docx.py` exits with code 0.

### P19: Heading fonts must use 宋体 for eastAsia

Heading 1 and Heading 2 styles, as well as `add_heading()`, must set
`eastAsia` to `'宋体'`. Do not use 黑体 for the eastAsia font setting.

```python
# CORRECT — headings use 宋体 (confirmed by format checker)
h1_rf.set(qn('w:eastAsia'), '宋体')
set_run_font_eastasia(run, '宋体')  # in add_heading

# WRONG — format checker will flag this
# h1_rf.set(qn('w:eastAsia'), '黑体')
```

### P20: AI-generated source IDs and figure URLs are unreliable

When downloading figures for Literature Review, never trust AI-generated source IDs,
repository IDs, article numbers, or figure URLs. They are frequently wrong and may point
to unrelated materials. Always verify the source through the DOI, publisher page, official
repository, or catalog record before using any figure.

```
# WRONG — use an AI-supplied source URL directly
curl "https://example.invalid/articles/unknown-source/figure1.png"  # might be wrong source

# CORRECT — verify source first
1. Open the DOI, publisher page, official repository, or catalog record
2. Confirm the title, authors/source organization, year, and figure caption
3. Then fetch figures from the verified source
```

### P21: Mandatory post-generation visual review of all figures

After generating the final document, use the `Read` tool to visually inspect EVERY
figure file, and verify:
1. The image content matches the section where it is inserted
2. The caption accurately describes what the image actually shows
3. The cited source paper is correct

This catches the most common and embarrassing error: a figure from source X inserted in
a section discussing topic Y because the source URL was wrong or the figure was mislabeled.

### P22: Ban Chinese "XX 等" filler and narrative "Author 等 YYYY" citations

Chinese "XX 等" usage has two distinct forms. Both should be avoided in formal thesis
body text unless they are part of a reference-list entry.

**Form A — list filler**: `"A、B、C、D 等指标"` / `"X、Y、Z 等工具"` / `"... 等多个维度"`.
Signals an incomplete list or AI padding. Replace with a closed list using exact count/scope:
`"A、B、C、D 四类指标"` or `"X、Y、Z 与 W 这组工具"` or list the dimensions explicitly.

**Form B — narrative author-etal citation**: `"Author 等 2024 的研究"` /
`"Author 等的后续工作"` / any `Author + 等` narrative reference. Replacing it with English
`Author et al.` usually still reads mechanical. Prefer a topical description plus `[N]`
citation:

- WRONG: `Author 等 2024 年的基准工作...`
- WRONG: `Author et al. 2024 年的基准工作...`
- CORRECT: `2024 年的一项基准工作...[32]`
- CORRECT: `早期基准研究...[32]`

Section headings follow the same rule. Avoid headings like `"4.2 与 Author et al. 2024
框架的对照"`; use `"4.2 与早期基准框架的对照"` or another topical heading.

**Exception**: reference list entries themselves use `author, author, et al.` — that is a
GB/T 7714 formatting requirement and must be preserved. When scanning, split the document
into body vs reference list (the list starts at the `[1] Xxx` entry under the 参考文献
heading) and scan only the body for `et al.` and narrative `XX 等`.

### P23: Academic thesis humanizing must stay formal and metaphor-free

The bundled `humanizer` skill includes a general prose mode that can add voice,
first-person phrasing, humor, or vivid imagery. That guidance does not apply to Chinese
undergraduate thesis writing. Academic thesis register should be formal, third-person,
precise, and free of decorative metaphors. Variety should come from clearer logic,
controlled sentence length, and exact terminology, not from imagery.

When running a humanization pass on a thesis, override any creative-writing guidance
explicitly in the worker prompt or local revision instructions:

```
HARD OVERRIDE (academic thesis style):
- No metaphors, no personification, no colloquialisms, and no entertainment-style imagery.
- No first-person (我 / 笔者 / 我们). Use 本文 / 该研究 / 结果表明 instead.
- No emotional descriptors, casual evaluation, or conversational commentary.
- No rhetorical flourishes, slogan-like phrasing, or ornamental synonyms.
- Replace vivid or colloquial wording with precise academic verbs and nouns.
- The humanizer skill's "Personality and Soul" section does NOT apply here — skip it.
```

Out-of-register creative phrasing can be as damaging as stiff AI phrasing. The safe
register for a Chinese undergraduate thesis is plain, precise, neutral, and
evidence-bound.

**Scope — both forms apply to headings, TOC entries, and body text.** Section titles can
also contain these errors, so verify them alongside normal paragraphs.

**Scanning protocol** (mandatory after every docx regeneration):

```python
# Extract all 等 contexts from document.xml, then classify:
# - LEGITIMATE ("不等同", "并不等价", "等于", "中等", "等级", etc. — "equal"/"grade" senses)
# - VIOLATION A (filler before a noun): "...等工具/指标/数据库/流程/维度/场景/类型/特征"
# - VIOLATION B (author-etal): "[A-Z][a-z]+\s*等" or "[A-Z][a-z]+\s*等\s*\d{4}"
# Any VIOLATION must be fixed before delivery.
```

Store the scan script pattern in the generator directory so it can be rerun on demand.
Do NOT trust a single pre-regeneration grep — the humanizer pass can re-introduce "等"
constructions (especially in author-et-al form) that weren't there before.
Always re-scan post-humanize with both filler and narrative author-etal forms included.

## 6. Forms

### P24: Task book and midterm-check support must not restore old private templates

Task book and midterm-check generation are supported, but do not reintroduce old
single-purpose template files or example content. Build clean table/form scripts for the
current user's document, using placeholders for missing personal fields and generic
academic wording. Use `references/forms.md` for the current form structure.

### P25: Midterm-check content is progress-level, not result-level

The midterm-check form should describe completed work, remaining work, practical problems,
and proposed solutions. Do not insert detailed results, named variables, private data, or
technical conclusions unless the user explicitly provides them and asks to keep them.
